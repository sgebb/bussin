Dead Letter Queue Debugging: A Complete Workflow

How to debug Azure Service Bus dead letter queues efficiently. Peek DLQ messages, inspect failure reasons, fix your code, and resubmit - all from the browser.

What Ends Up in the Dead Letter Queue?

Messages land in the DLQ for several reasons:

  • Max delivery count exceeded - the consumer threw an exception too many times
  • TTL expiration - the message sat in the queue longer than its time-to-live
  • Filter evaluation errors - a subscription filter couldn't evaluate the message
  • Manual dead-lettering - your application explicitly dead-lettered the message

Whatever the cause, the messages are sitting in the DLQ with their original bodies and properties intact, plus extra dead-letter metadata like DeadLetterReason and DeadLetterErrorDescription.

Step 1: Toggle DLQ View

In Bussin, select your queue or topic subscription, then click the DLQ toggle button in the actions sidebar. The message counts update immediately - you'll see how many dead-lettered messages are waiting.

Step 2: Peek DLQ Messages

Use Peek or Advanced Peek to inspect the messages. Since these are dead-lettered, they won't be consumed - peeking is always safe. You can peek up to 10,000 DLQ messages at once using Bussin's iterative peeking.

Once loaded, use the inline filter to search by message body content, MessageId, or Subject. If your DLQ has more than 10,000 messages, use Deep Search mode to scan up to 1 million messages in the background.

Step 3: Inspect Message Details

Click any message to open the detail modal. You'll see:

  • Broker properties - Sequence number, enqueued time, delivery count, TTL, content type, correlation ID, session ID, partition key, and expiry time
  • Custom (application) properties - all key-value pairs your producer attached
  • Message body - auto-formatted as JSON if applicable, with a copy-to-clipboard button

The delivery count is especially useful - if it's at the max delivery count for the entity (usually 10), you know the message was dead-lettered due to repeated processing failures.

Step 4: Resubmit or Clean Up

Once you've fixed the underlying bug, you have two options:

  • Resubmit (single) - Click Resubmit in the message detail modal to move one message back to the active queue
  • Resubmit All - Click Resubmit All in the sidebar to move every DLQ message back. This runs as a background task so you can continue working.

If the messages are stale and no longer needed, use Purge to clear the entire DLQ. Bussin will ask for confirmation first.

Step 5: Move Messages to DLQ

Sometimes you need to move active queue messages to the DLQ - for example, to quarantine a batch of poison messages that are causing consumer crashes. In Bussin, open a message from the active queue and click Move to DLQ.

Receive & Lock for Granular Control

For more precise operations, use the Receive & Lock feature. Lock up to 100 messages at once, inspect them, and then choose per-message: Complete (delete), Abandon (release lock), or Dead Letter. This is ideal for selectively cleaning up a queue without affecting healthy messages.

(Warning) Session-enabled entities
Resubmit, purge, and single-message delete operations are not supported for session-enabled entities in the current version. This is a limitation of the peek-lock model when sessions are involved.
