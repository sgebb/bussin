Deep Search: Scouring Millions of Messages in the Browser

Learn how Bussin bypasses standard peeking limits to scan up to 1 million messages in the background without locking your UI.

Standard peeking in Azure Service Bus is limited, often capping out around 10,000 messages. But what if your Dead-Letter Queue has 500,000 messages and you need to find one specific trace ID? That's where Deep Search comes in.

How Deep Search Works

When you add a search filter (Message ID, Subject, or Body content) in Advanced Peek and the entity has more than 10,000 messages, Bussin automatically suggests Deep Search Mode. This launches an intelligent search as a background task that iterates through up to 1 million messages to find your match.

  1. Open Advanced Peek (the gear icon (GearIcon) next to Peek)
  2. Enter a filter - for example, a MessageId or a specific "customerId": 123 snippet in the JSON body
  3. If the entity has >10K messages, you'll see the Deep Search option prompted automatically
  4. Click Start Deep Search

Tracking progress

Unlike old desktop tools that lock up your interface while they poll APIs, Bussin runs the search fully in the background using a web worker model. You can:

  • See real-time scanning progress in the Running Tasks panel (bottom-left)
  • Continue using other parts of the app - peek other queues, send messages, or organize your namespace
  • Cancel the search gracefully at any time
  • View results instantly when it completes - Bussin shows the matching sequence numbers and provides direct links to jump to those payloads

Filter options

All three search filters use case-insensitive matching locally in your browser memory for maximum speed:

  • Message ID: Exact or partial match on the broker-assigned or custom message ID
  • Subject: Search the Subject/Label metadata field
  • Body Content: Search anywhere inside the raw message body string
(Info) Practical example
Your producer sends order events with a payload containing "CorrelationId": "ORD-999". A customer reports their order wasn't processed. Use Deep Search with a body filter for "ORD-999" to scan the massive Dead Letter Queue, verify its content, and instantly determine exactly why it failed based on its delivery count or metadata error reason.
