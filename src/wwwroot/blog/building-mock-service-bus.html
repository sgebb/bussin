I Built a Fake Azure Service Bus (and It Was Worth Every Hack)

How I wrote an in-process AMQP mock broker so my browser-based Service Bus tool could run a demo mode and CI tests without touching Azure.

I've been building Bussin, a browser-based Azure Service Bus explorer, for a while now. It talks AMQP directly from the browser to Azure â€” no backend, no proxies. That architecture is great for security and simplicity, but it meant I had essentially zero automated tests and a demo mode held together with duct tape. Every time I changed something in the AMQP layer, I was just crossing my fingers and deploying.

The Problem: Testing Against a Cloud Service You Don't Control

Bussin is a Blazor WebAssembly app that speaks raw AMQP via the rhea library, over WebSockets, directly to Azure Service Bus. There is no backend. That's the architecture â€” but it also means every test needs a live Service Bus namespace.

My CI pipeline on GitHub Actions would have needed Azure credentials in a secret, a running namespace with pre-seeded data, and a stable network connection. Locally, I had to be logged in with the right tenant and have messages already sitting in queues. It was fragile and slow. I had no real tests, and I was increasingly nervous about shipping regressions.

I also had an existing demo mode â€” a C# class (DemoServiceBusService) that implemented the same IServiceBusJsInteropService interface as the real one, returning hardcoded fake data. It worked for showing the UI, but it was completely disconnected from the actual AMQP code path. The demo mode didn't test anything real. If I broke the message parser or the management client, the demo would still work fine and I'd only discover the bug when a user hit it against a real namespace.

I looked at the Azure Service Bus Emulator â€” the Docker-based local emulator that Microsoft ships. It's a serious piece of software, but it pulls about a gigabyte of container images (it needs SQL Server as a dependency), only supports TCP (no WebSockets), and doesn't support Entra ID auth. For a browser app that speaks AMQP over WebSockets, it was a non-starter for the demo mode. And spinning up a gig of Docker infrastructure in a CI runner for a handful of message tests felt disproportionate.

The Idea: Fake the Broker, Not the API

The insight that made this work was realising I didn't need to mock the Azure SDK. Bussin doesn't use the Azure SDK. It speaks raw AMQP via rhea. So the mock didn't need to pretend to be an Azure client â€” it needed to pretend to be a Service Bus broker.

The key design decision: intercept at the connection level, not the API level. My ServiceBusConnection class already had a single point where it calls rhea.connect(). If I replaced that one call with a mock connection that implemented the same EventEmitter-based interface that rhea returns, the entire 1,600-line API layer above it would work unchanged.

// connection.ts â€” the entire mock integration point\nif ((globalThis as any).__BUSSIN_SIMULATOR_ACTIVE__) {\n    this.connection = GlobalMockBroker.connect({\n        container_id: `bussin-${this.namespace}`\n    }) as any as Connection;\n    resolve();\n    return;\n}

One global boolean. One if statement. That's the only change to the production connection code.

What the MockBroker Actually Does

The MockBroker is about 450 lines of TypeScript. It simulates enough of the AMQP management protocol to fool the rest of the application into thinking it's talking to Azure. Here's what it handles:

  • Topology: createQueue(), createTopic(), createSubscription() â€” you define the shape of your namespace up front. Topics automatically fan out to all subscriptions when you push a message.
  • CBS authentication: The Claims-Based Security handshake that Azure requires. The mock just returns a 202 Accepted for any token â€” because in demo mode, there is no real auth.
  • Management operations: com.microsoft:peek-message, com.microsoft:receive-by-sequence-number, com.microsoft:update-disposition, com.microsoft:purge-messages. These are the actual AMQP management operations that the Azure SDK uses under the hood. The mock parses the incoming management request, manipulates its in-memory queues, and returns a correctly shaped response.
  • Sequence numbers and lock tokens: Each message gets a monotonic sequence number. Receive-and-lock generates fake 16-byte lock tokens. Dispose uses those tokens to complete, abandon, or dead-letter.
  • Dead Letter Queues: When a message is \"suspended\" (Azure's term for dead-lettering via management), it gets moved to a $deadletterqueue path under the entity. Properties are preserved.

Two Wins: Demo Mode and CI

Demo Mode

The biggest payoff was replacing the old C# demo service. Now when you visit Bussin and click \"Run Demo\", the Blazor app registers a DemoServiceBusJsInteropService that calls enableSimulator(true), seeds the mock broker with realistic data (two namespaces, queues with orders, a topic with inventory and billing subscriptions), and then every subsequent API call â€” peek, send, purge, resubmit, dead-letter â€” goes through the mock and through the real AMQP code path.

Every operation works. The message counts update, dead-lettered messages appear in the right place, purging clears the queue. It's the same UI, the same code paths, the same AMQP message parsing. The only difference is the bytes never leave the browser. And crucially, if I break the message parser, the demo breaks too â€” so I'll notice before anyone else does.

CI/CD Tests

The second win was CI. I now run test suites against the mock broker in GitHub Actions:

- name: Build and Test client-js\n  run: |\n    cd client-js\n    npm ci --ignore-scripts\n    npm test\n    npm run build

No Azure credentials. No Docker. No network. The tests run in under 2 seconds and cover:

  • Topology fidelity: Send a message to a topic, verify it fans out to all subscriptions. Dead-letter from one subscription, verify the other subscription is unaffected.
  • Stability: Batch send 3 messages, verify all 3 are stored. Receive-and-lock, then dead-letter, verify the message moved correctly.
  • Resubmit workflows: Push a message to DLQ, peek it, send a modified copy back to the topic, complete the DLQ copy, verify properties survived the round-trip.

These are the kinds of operations I was most nervous about breaking. The resubmit test, for instance, caught a bug where message properties were being dropped during the DLQ-to-active-queue workflow â€” something that would have been very annoying to discover in production.

The Shortcomings (Honest Assessment)

Let me be upfront about what this is not: it's not a general-purpose Azure Service Bus emulator. It's a mock that's shaped exactly like the subset of AMQP operations that Bussin uses. The gaps are real:

  • No subscription filters. Every message goes to every subscription. There's no SQL filter or correlation filter evaluation.
  • No sessions. Session-enabled queues aren't simulated. The group_id property is preserved but has no ordering semantics.
  • No TTL enforcement. Messages never expire. There's no background timer to dead-letter expired messages.
  • No max delivery count. You can abandon a message infinitely without it auto-dead-lettering.
  • No scheduled delivery. Messages are immediately available regardless of any scheduled enqueue time.
  • No transactions. There's no concept of a transaction scope.
  • Sequence number semantics are simplified. Each entity has its own counter, but there's no concept of deferred messages or sequence number gaps.

And then there's the deeper concern: am I actually testing real things? The mock introduces its own complexity. When a test passes, is that because my application logic is correct, or because my mock is conveniently shaped to make it pass? The mock doesn't enforce any of the subtle timing, ordering, or error semantics that a real broker does. A test can pass against the MockBroker and still fail against Azure because the real broker behaves differently under load, with concurrent consumers, or with network partitions. I'm aware of this trade-off. The tests catch regressions in my code â€” message parsing, property mapping, the disposition state machine â€” but they're not a substitute for testing against the real thing before a release.

For Bussin's use case â€” a management UI that peeks, sends, locks, dead-letters, resubmits, and purges â€” these gaps are acceptable. For testing a consumer application that depends on subscription filters or session ordering, they wouldn't be.

Could You Use This as a Mock Service Bus for .NET?

The short answer is: not directly, but the approach is portable.

The MockBroker is written in TypeScript and implements rhea's EventEmitter-based connection interface. If your .NET application uses the official Azure.Messaging.ServiceBus SDK, it doesn't speak rhea â€” it uses the Microsoft.Azure.Amqp library underneath. You'd need a different interception point.

However, the strategy translates well. If you're testing at the integration level, you could:

  1. Use the official emulator. The Service Bus Emulator works with the .NET SDK over TCP with SAS auth. It needs Docker + SQL, but it's the most faithful option.
  2. Mock at the SDK level. The ServiceBusProcessor and ServiceBusReceiver classes aren't easily mockable, but you can abstract behind an interface and swap implementations.
  3. Port the MockBroker concept. If you're building a tool that speaks raw AMQP (like I was), the same approach works: intercept at the transport layer, return correctly shaped AMQP responses, and simulate the management operations you need.

If someone wanted to turn this into a standalone mock server, the interesting path would be to make the MockBroker listen on an actual WebSocket (or TCP) port and speak real AMQP frames. At that point, any AMQP client in any language could connect to it. That's a significantly bigger project, but the state management and topology logic in the current MockBroker would transfer directly.

What I Actually Learned

Building this taught me more about how Azure Service Bus works internally than years of using it through the SDK. The management protocol â€” the com.microsoft:peek-message operations, the lock token lifecycle, the disposition state machine â€” is well-documented but rarely seen directly. Implementing it from scratch gave me a much better mental model of what's actually happening when I call receiver.CompleteMessageAsync().

It also validated a design philosophy I keep coming back to: if your application has a single integration point with an external service, that point is your mock boundary. I didn't need to mock 50 API functions. I mocked one connection factory, and 1,600 lines of application code worked unchanged.
