Auto-Forwarding: Azure Service Bus's Most Under-Appreciated Feature

Why giving your downstream teams true ownership of their queues is the secret to scalable, drama-free message routing.

When working with Azure Service Bus Topics, a common architectural pattern is simple: the publisher sends an event, and the consumer's worker process pulls directly from a Subscription to process it. While this works, it creates a massive operational bottleneck known as the "Fat Subscriber" problem.

The Core Issue: Who Owns What?

When you read directly from a Subscription, the downstream consumer team is relying completely on the publisher's infrastructure. This leads to several massive headaches:

  • Cross-Team Alarms: The lowest level entity within a Topic is the Topic itself for most alerting. If a downstream consumer breaks and their Subscription Dead-Letter Queue (DLQ) fills up, the publisher's on-call engineer often gets the alarm.
  • Broadcast Nightmares: If an event fails and goes to the DLQ, how do you replay it? If you resubmit it to the Topic, it gets broadcasted to all subscribers again, causing other healthy consumers to process duplicates.
  • Constrained Permissions: This is the big one. To do a "resend" (e.g., resubmitting a dead-lettered message or doing an exponential backoff), you need Send permissions. You cannot grant Send permissions on a Subscriptionâ€”they must be granted on the whole Topic. Furthermore, Subscription-level RBAC isn't even visible or editable in the Azure Portal GUI, making it a nightmare to manage.

The Solution: Auto-Forwarding to Dedicated Queues

The solution is Subscription-to-Queue auto-forwarding. The architecture changes conceptually: The publisher owns the Topic, and the subscriber owns the Queue.

Instead of the worker reading from the Subscription, the Subscription acts as a dumb routing filter that instantly auto-forwards all matched messages into a dedicated Queue owned by the downstream consumer team.

The Strategic Benefits

  1. True Permissions Independence: The consumer team can be granted "Data Owner" access to their specific Queue. This means they can implement custom logic like exponential backoffs directly onto their Queue without needing broad permissions on the publisher's Topic.
  2. Granular, Team-Specific Alarms: Queue depth, DLQ size, and latency alarms are scoped exactly to the consumer's Queue. The alarms bypass the publisher entirely and page the correct downstream team.
  3. Safe Replays: If a consumer team fixes a bug, they can resubmit dead-lettered messages directly back onto their Queue. The message is processed successfully without being broadcast to any other subscriber.

Writing the Infrastructure as Code

Implementing this pattern is extremely easy in both Bicep and Terraform.

Bicep Implementation

param topicName string = 'events-topic'
param queueName string = 'consumer-a-queue'
param subscriptionName string = 'sub-consumer-a'

// The Consumer's Queue
resource queue 'Microsoft.ServiceBus/namespaces/queues@2022-10-01-preview' = {
  name: queueName
  parent: serviceBusNamespace
}

// The Topic Subscription (Filter & Router)
resource subscription 'Microsoft.ServiceBus/namespaces/topics/subscriptions@2022-10-01-preview' = {
  name: subscriptionName
  parent: topic
  properties: {
    // The magic setting:
    forwardTo: queue.name
  }
}

Observability & Debugging

One tricky aspect of auto-forwarding is debugging. If auto-forwarding fails (for example, if the destination queue is deleted or reaches its maximum size), messages are sent to the Subscription's Dead Letter Queue, with a `DeadLetterReason` of `TransferEndpointUpdateFailed`.
